This track is for **people working on Calimero Core itself** — the node daemon,
the runtime, the sync engine, the SDK, and the surrounding tooling. If you want
to _build an app_ on Calimero, or _operate a node_, those are separate tracks.
If you want to _reimplement the protocol_ from first principles, read the
[Protocol Reference](/protocol/overview/) instead — it describes the wire and
state model independent of this codebase.

## What the repo is

Calimero Core is a **Rust monorepo** — a single Cargo workspace
(`resolver = "2"`) holding roughly **three dozen library and binary crates**
under `crates/`, plus example WASM applications under `apps/` and developer
utilities under `tools/`.

The shape, at a glance:

- **`crates/`** — the core libraries and the two binaries (`merod`, the node
  daemon; `meroctl`, the CLI). This is where almost all the work happens.
- **`apps/`** — example and test applications compiled to
  `wasm32-unknown-unknown` (e.g. `kv-store`), including a large suite of
  migration scenarios. Most are excluded from workspace versioning.
- **`tools/`** — standalone operator/developer CLIs (`calimero-abi`,
  `mero-sign`, `merodb`) for ABI inspection, bundle signing, and database
  debugging.

The system runs as a set of **Actix actors** over a **WASM runtime** (Wasmer),
a **libp2p** P2P stack, and a **RocksDB**-backed storage layer. The next page
walks that architecture crate by crate.

<Aside type="note">
  This track documents the codebase as it exists today. Some crates are
  mid-migration (notably the unified causal-log work under `op` / `op-adapter` /
  `projection` / `authz`); those are flagged where they appear.
</Aside>
